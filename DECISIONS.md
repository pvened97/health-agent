# Architecture Decision Records — Health Agent

## ADR-001: APScheduler вместо Celery для фоновых задач

**Дата:** 2026-03-12
**Статус:** принято

**Контекст:** Нужен планировщик для: утренних сообщений, weekly summary, WHOOP token refresh, periodic sync.

**Решение:** APScheduler (AsyncIOScheduler) встроен в FastAPI-процесс.

**Аргументы за:**
- Не нужен Redis и отдельный worker-процесс
- Достаточен для single-user MVP
- Простота деплоя (один контейнер)
- Нативно async

**Аргументы против:**
- Не масштабируется горизонтально
- Задачи теряются при рестарте (нет persistence)

**Пересмотреть когда:** multi-user, тяжёлые фоновые задачи, несколько инстансов.

## ADR-002: PostgreSQL вместо SQLite

**Дата:** 2026-03-12
**Статус:** принято

**Контекст:** Нужна БД для persistent state.

**Решение:** PostgreSQL 16 в Docker.

**Аргументы за:**
- jsonb для метрик и аргументов tool calls
- Полноценные транзакции и concurrent access
- Готовность к production
- Partial unique indexes (для soft delete + unique constraints)

**Аргументы против:**
- Требует Docker для dev-окружения

## ADR-003: Polling для dev, webhook для production (Telegram)

**Дата:** 2026-03-12
**Статус:** принято

**Контекст:** Telegram Bot API поддерживает polling и webhooks, они взаимоисключающие.

**Решение:** Конфигурируемый режим через env-переменную. Polling по умолчанию.

**Аргументы за:**
- Polling не требует HTTPS и публичного IP для разработки
- Webhook — стандарт для production (мгновенная доставка, меньше нагрузки)

## ADR-004: Key-value модель для user_profile

**Дата:** 2026-03-12
**Статус:** принято

**Контекст:** Профиль пользователя содержит разнородные факты: цель, вес, аллергии, предпочтения.

**Решение:** Таблица user_profile с category/key/value вместо фиксированных колонок.

**Аргументы за:**
- Гибкость: новые типы фактов без миграций
- Агент может записывать произвольные факты
- Удобно для listing/management ("покажи что ты обо мне знаешь")

**Аргументы против:**
- Нет type safety на уровне БД
- Сложнее делать выборки по конкретным полям

**Митигация:** Валидация на уровне приложения. Для часто используемых фактов (вес, цель) — константы с ключами.

## ADR-005: OpenAI GPT-5.4 как основная модель

**Дата:** 2026-03-12
**Статус:** принято

**Контекст:** Выбор LLM для агента.

**Решение:** GPT-4o для MVP. Переход на двухуровневую схему (GPT-4o-mini + GPT-5.4) в итерации 8.

**Этапы:**
1. Итерации 1-7: GPT-4o — единая модель для всех задач
2. Итерация 8: split на fast (4o-mini) и strong (5.4) с роутингом по типу задачи

**Аргументы за поэтапный переход:**
- Сначала отладим tools и логику на одной модели
- В итерации 7 появятся traces и evals — сможем замерить качество при смене модели
- Не оптимизируем стоимость до того, как поймём реальный usage pattern

## ADR-006: Single-agent архитектура для MVP

**Дата:** 2026-03-12
**Статус:** принято

**Контекст:** Можно сделать несколько специализированных агентов (тренер, нутрициолог, аналитик) или одного универсального.

**Решение:** Один агент с полным набором tools. Multi-agent — в v2.

**Аргументы за:**
- Проще отладка
- Меньше latency (нет handoffs)
- Достаточно для MVP

**Пересмотреть когда:** system prompt станет слишком длинным, качество ответов упадёт из-за смешения ролей.

## ADR-007: Two-phase memory (запись → consolidation)

**Дата:** 2026-03-12
**Статус:** принято

**Контекст:** Агент не должен записывать ложные факты в устойчивую память после одного события.

**Решение:**
1. События → журнал (всегда)
2. Паттерны → memory_notes (кандидаты, agent_inferred)
3. Кандидаты → profile/derived_rules (только с подтверждением)

Memory distillation — отдельный tool, не автоматическая запись.
