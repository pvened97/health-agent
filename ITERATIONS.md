# Plan итераций Health Agent

## Iteration 1 — Скелет проекта

**Цель:** запускаемое приложение с минимальным end-to-end flow.

- [ ] Инициализация Python-проекта: pyproject.toml / requirements.txt
- [ ] `.env.example` с переменными (OPENAI_API_KEY, TELEGRAM_BOT_TOKEN, DATABASE_URL)
- [ ] `app/config.py` — Pydantic Settings, чтение .env
- [ ] `docker-compose.yml` — PostgreSQL 16
- [ ] `app/database.py` — SQLAlchemy async engine + session factory
- [ ] Alembic init + первая миграция
- [ ] Модели SQLAlchemy: `users`, `telegram_accounts` (минимум)
- [ ] `app/main.py` — FastAPI app с health-check endpoint
- [ ] `app/telegram/bot.py` — базовый бот, polling mode, echo-ответ
- [ ] `app/telegram/handlers.py` — handler входящих сообщений, маршрутизация на агента
- [ ] `app/agent/agent.py` — OpenAI Agent с пустым system prompt и одним dummy tool
- [ ] Проверка: отправил сообщение в Telegram → бот передал агенту → агент ответил

**Что ты получишь как пользователь:**
Ты открываешь Telegram, пишешь боту любое сообщение — и получаешь осмысленный ответ от GPT-5.4. Бот пока ничего не запоминает и не логирует, но весь pipeline работает: Telegram → FastAPI → OpenAI Agent → ответ. PostgreSQL поднят в Docker, структура проекта готова к расширению. Ты можешь убедиться, что всё запускается одной командой и бот живой.

**Цель:** полная схема БД и tools для логирования.

- [ ] Модели SQLAlchemy (все сущности из требований):
  - `user_profile` (цель, ограничения, предпочтения, антропометрия)

  - `meal_logs` (время, продукты, калории, белок, комментарий, source)
  - `workout_logs` (тип, длительность, интенсивность, комментарий, source)
  - `recovery_logs` (score, hrv, rhr, source, external_id)
  - `body_metrics` (вес, % жира, замеры)
  - `daily_notes` (свободный текст, настроение, самочувствие)
  - `memory_notes` (промежуточная память до консолидации)
  - `derived_rules` (гипотезы, confidence, status, confirmation)
  - `agent_runs`, `tool_calls` (observability)
- [ ] Alembic миграции для всех таблиц
- [ ] Soft delete (deleted_at) на логах и памяти
- [ ] Source enum: user_manual, whoop_api, agent_inferred, system_aggregated
- [ ] Tools:
  - `get_user_profile` — читает профиль
  - `save_sleep_log` — сохраняет запись сна
  - `save_meal_log` — сохраняет запись питания
  - `save_workout_log` — сохраняет запись тренировки
  - `save_note` — сохраняет заметку/самочувствие
  - `get_recent_logs` — последние N записей по типу
- [ ] Idempotency: дедупликация по external_id для повторных событий

**Что ты получишь как пользователь:**
Ты пишешь боту естественным языком — "съел 200г куриной грудки с рисом", "спал 7 часов, проснулся в 7:30", "сделал силовую 45 минут, ноги" — и агент понимает, что это за тип события, структурирует данные (калории, белок, длительность, тип нагрузки) и сохраняет в базу. Ты можешь спросить "что я ел сегодня?" или "какие тренировки были на этой неделе?" — и получить ответ на основе реальных записей из БД, а не из истории чата. Данные не теряются между сессиями.

## Iteration 3 — Memory system

**Цель:** 4-слойная память с consolidation и management.

- [ ] `app/agent/context.py` — selective memory injection:
  - при каждом запросе собирает: профиль + текущее состояние + релевантные недавние логи
  - НЕ загружает всю историю
- [ ] `app/agent/prompts.py` — динамическая сборка system prompt из контекста
- [ ] Tools для памяти:
  - `get_current_state` — агрегаты (средний сон за 7 дней, последний recovery, тренировки за неделю)
  - `save_derived_rule` — сохранить гипотезу с confidence
  - `list_memory_candidates` — показать pending memory notes
  - `promote_memory_candidate` — перенести в профиль (с подтверждением пользователя)
  - `delete_memory_item` — удалить факт
  - `update_user_profile` — обновить устойчивый факт (через подтверждение)
- [ ] Consolidation logic:
  - одноразовые события → журнал
  - повторяющиеся паттерны → memory_notes (кандидаты)
  - подтверждённые кандидаты → profile / derived_rules
- [ ] Конфликт-резолвинг: при обновлении профиля спрашивать подтверждение
- [ ] Команды пользователя: "покажи мой профиль", "удали факт X", "что ты обо мне знаешь"

**Что ты получишь как пользователь:**
Агент знает тебя. Ты один раз сказал "мне 30 лет, вешу 82 кг, цель — набор мышечной массы, не ем глютен" — и он это помнит всегда, в каждом следующем разговоре. Ты можешь сказать "покажи, что ты обо мне знаешь" — и увидеть список фактов. Можешь сказать "удали факт про глютен" или "я теперь вешу 80" — и агент обновит память, предварительно попросив подтверждение. Если агент заметит паттерн (например, ты 5 раз подряд написал, что пьёшь кофе после 18:00), он предложит это зафиксировать как наблюдение, но не запишет автоматически — только после твоего ОК. Контекст подгружается умно: агент не тащит всю историю, а берёт только релевантное.

## Iteration 4 — Рекомендации и аналитика

**Цель:** ежедневные рекомендации и weekly summary.

- [ ] Tools:
  - `get_daily_recommendation_context` — собирает все данные для рекомендации на день
  - `get_week_summary` — агрегирует данные за 7 дней
- [ ] `app/quality/rules.py` — детерминированные правила:
  - нет recovery → не рекомендовать уверенно
  - сон < 6ч 3 дня → повышенная осторожность
  - смена цели → пересчёт с новой даты
  - валидация калорий (не может быть 50000 ккал)
  - приоритет источников (WHOOP > manual для sleep metrics)
- [ ] Ежедневная рекомендация: агент учитывает recovery + сон + тренировки + питание
- [ ] Weekly summary: обзор за 7 дней, паттерны, выводы
- [ ] Объяснение рекомендации: по запросу агент показывает данные, на которых основан вывод
- [ ] Disclaimer при высокорисковых рекомендациях

**Что ты получишь как пользователь:**
Ты спрашиваешь "как мне сегодня тренироваться?" — и агент не выдумывает ответ, а идёт в базу: смотрит твой recovery, сколько ты спал, когда была последняя тяжёлая тренировка, что ел. И отвечает: "Recovery 68%, сон 6.5ч — ниже твоей нормы. Последняя тяжёлая тренировка была вчера. Рекомендую лёгкое кардио или день отдыха." Ты можешь спросить "почему?" — и он покажет конкретные цифры. Если данных мало, он скажет это прямо, а не будет выдумывать. Раз в неделю ты можешь попросить сводку — и получить обзор: средний сон, количество тренировок, тренд recovery, что можно улучшить. Система не даст опасных рекомендаций — есть жёсткие правила в коде, которые перестраховывают модель.

## Iteration 5 — Level Kitchen: каталог блюд

**Цель:** автоматический подсчёт калорий для блюд из Level Kitchen.

- [ ] Таблица `meal_catalog` в БД:
  - name (название блюда)
  - calories, protein_g, carbs_g, fat_g
  - meal_type (breakfast/lunch/dinner/snack)
  - source_url
  - week_start (дата начала недели меню)
  - parsed_at
- [ ] Парсер меню Level Kitchen:
  - скрипт, который заходит на сайт и извлекает блюда с БЖУ
  - запуск по расписанию раз в неделю (scheduler)
  - сохранение в `meal_catalog` с дедупликацией
- [ ] Tool `search_meal_catalog`:
  - агент вызывает его когда пользователь упоминает блюдо из Level Kitchen
  - нечёткий поиск по названию (пользователь может написать не точно)
  - возвращает название, калории, БЖУ
- [ ] Обновление промпта:
  - если пользователь упоминает Level Kitchen или блюдо похоже на позицию из каталога — сначала поискать в каталоге, потом записывать
- [ ] Fallback: если блюдо не найдено в каталоге — оценить вручную как обычно

**Что ты получишь как пользователь:**
Ты пишешь "съел на обед курицу терияки от Level Kitchen" — агент находит это блюдо в каталоге и записывает точные калории и БЖУ с сайта, а не свою оценку. Меню парсится автоматически раз в неделю, когда Level Kitchen публикует новое. Если блюдо не нашлось — агент оценит сам и скажет, что это приблизительно.

**Цель:** подключение WHOOP, ручная и автоматическая синхронизация данных с локала.

- [x] `app/whoop/oauth.py` — OAuth 2.0 flow:
  - authorization URL
  - callback endpoint в FastAPI
  - сохранение access + refresh token
  - offline scope
- [x] `app/whoop/client.py` — WHOOP API client:
  - get sleep, recovery, workouts
  - обработка 401 с auto retry
  - auto token refresh
- [x] `app/whoop/sync.py` — синхронизация:
  - сохранение в БД с external_id
  - дедупликация
  - маппинг WHOOP → локальные модели (sleep_logs, recovery_logs, workout_logs)
- [x] `whoop_connections` + `sync_events` модели
- [x] Tools агента:
  - `get_whoop_status` — статус подключения
  - `sync_whoop_now` — ручной запуск синхронизации
  - `get_latest_whoop_metrics` — последние метрики (recovery, сон, тренировки)
- [x] Команда `/whoop` в Telegram: подключить или синхронизировать
- [x] Context injection: последний recovery с цветовой зоной
- [x] Промпт агента с правилами по WHOOP (зоны recovery, когда какой tool вызывать)

**Что ты получишь как пользователь:**

## Iteration 7 — Деплой на VPS + WHOOP webhooks + Scheduler

**Цель:** перенести бота на виртуалку, добавить WHOOP webhooks для реалтайм-данных, настроить scheduler.

- [x] **Деплой на VPS:**
  - Dockerfile для приложения
  - docker-compose.yml: app + PostgreSQL
  - Telegram polling mode (webhook готов, но polling достаточно для MVP)
  - HTTPS через Caddy + Let's Encrypt + DuckDNS (health-bot.duckdns.org)
  - Переменные окружения для prod
  - WHOOP OAuth работает через HTTPS
- [ ] **WHOOP webhooks:**
  - `app/whoop/webhook.py` — обработка входящих POST от WHOOP
  - Регистрация webhook через WHOOP API (одноразово при подключении)
  - Обработка событий: `recovery.updated`, `sleep.updated`, `workout.updated`
  - По событию: забираем полные данные по ID через API, сохраняем в БД
  - Идемпотентная обработка (повторный webhook не создаёт дубли)
  - Опционально: Telegram-уведомление при новых данных ("Новый recovery: 72% 🟢")
- [ ] **Scheduler (APScheduler):**
  - WHOOP token refresh (каждый час)
  - Nightly reconciliation sync (на случай пропущенных webhooks)
- [ ] **Proactive messages:**
  - Утренний check-in: "Доброе утро! Recovery X%, рекомендация на день..."
  - Weekly summary (воскресенье)
  - Настройки: время утреннего сообщения, вкл/выкл

**Что ты получишь как пользователь:**
Бот работает 24/7 на сервере. Данные с WHOOP приходят автоматически в реалтайме — ты снял браслет утром, и через минуту бот уже знает твой recovery и стадии сна. Утром приходит сообщение с рекомендацией на день без запроса. В воскресенье — недельный отчёт. Не нужно вручную синхронизировать или держать ноутбук включённым.

## Iteration 8 — Observability и качество

**Цель:** полная наблюдаемость и тестовое покрытие.

- [x] `agent_runs` — запись каждого вызова агента (input, output, model, duration)
- [x] `tool_calls` — запись каждого tool call (name, args, result, duration)
- [ ] Traces через OpenAI Agents SDK
- [x] Structured logging (JSON logs в prod, human-readable в dev)
- [x] Error handling:
  - fallback при ошибке модели (дружелюбное сообщение пользователю)
  - ошибка записи в БД не ломает основной flow
  - retry policy для OpenAI API
- [ ] Тесты:
  - unit: tools, quality rules, memory logic
  - integration: Telegram handler → agent → DB
  - idempotency: повтор update, повтор webhook
  - memory: событие не становится фактом без подтверждения
  - e2e: дневной check-in, weekly summary, обновление профиля
- [ ] Evals: набор эталонных сценариев для проверки качества ответов

**Что ты получишь как пользователь:**
Для тебя внешне ничего не изменится — но под капотом появляется страховка. Если агент дал странный ответ, ты (как разработчик) можешь посмотреть: какие tools он вызвал, какие данные получил, почему принял такое решение. Если OpenAI API упал — бот не зависнет, а скажет "сейчас не могу ответить, попробуй позже". Тесты гарантируют, что обновления кода не ломают рабочие сценарии.

## Iteration 9 — Двухуровневая модель (GPT-5.4-mini + GPT-5.4)

**Цель:** разные модели для разных задач — дешевле, быстрее, качественнее.

- [x] `app/config.py` — два параметра модели: `OPENAI_MODEL_FAST` (gpt-5.4-mini) и `OPENAI_MODEL_STRONG` (gpt-5.4)
- [x] Разделение задач по моделям:
  - **Fast (GPT-5.4-mini):** парсинг и запись (еда/сон/тренировка/заметка), запросы данных, фото еды, WHOOP статус/синхронизация
  - **Strong (GPT-5.4):** рекомендации, советы, планирование, аналитика, недельные обзоры, объяснения
- [x] `app/agent/router.py` — keyword-based роутер (27 паттернов, без LLM-вызова)
- [x] Debug mode: модель отображается в ответе `_[model: ...]_` (убрать после тестов)
- [x] Мониторинг: в `agent_runs` записывается какая модель использовалась
- [ ] A/B сравнение через evals: замерить качество fast vs strong на типовых сценариях

**Что ты получишь как пользователь:**
Простые действия ("записал завтрак", "вот что ты ел сегодня") станут быстрее и дешевле — их обрабатывает лёгкая модель. А когда ты спрашиваешь "как мне спланировать неделю тренировок с учётом моего recovery?" — подключается мощная модель, которая глубже анализирует данные. Ты не замечаешь разницы в интерфейсе, но получаешь лучший баланс скорости и качества.
