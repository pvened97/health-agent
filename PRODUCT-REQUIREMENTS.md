# Product Requirements — Health Agent

## 1. Назначение

Персональный цифровой ассистент по питанию, тренировкам, сну и восстановлению. Агентное приложение с собственным state layer, а не чат-бот с памятью в переписке.

Ключевые функции:
- Фиксация и накопление персонального контекста
- Автоматический импорт данных из WHOOP
- Структурированная история сна, восстановления, тренировок, питания
- Персональные рекомендации на основе данных с объяснением логики
- Управление памятью: просмотр, редактирование, удаление фактов

## 2. Бизнес-цели

- Замена встроенного WHOOP Coach на кастомного ассистента с лучшей персонализацией
- Объединение объективных данных (WHOOP) и субъективных (питание, самочувствие, заметки) в одном интерфейсе
- Накопление полезной памяти с контролем и редактированием
- Повышение качества повседневных решений: нагрузка, сон, питание, интерпретация recovery
- Архитектурная база для расширения: multi-agent, голос, веб-панель, другие источники данных

## 3. Ограничения продукта

- Не медицинский сервис
- Не ставит диагнозы
- Не даёт медицинских назначений
- Высокорисковые рекомендации — только с disclaimer и опорой на данные
- Переписка — не единственный источник истины

## 4. Scope MVP

### Входит в MVP

- Telegram-бот, один пользователь
- Backend на OpenAI Agents SDK
- PostgreSQL для persistent state
- Интеграция с WHOOP API
- Логирование: сон, recovery, тренировки, питание, самочувствие, заметки
- Ежедневный анализ состояния
- Недельная сводка
- Управление памятью: просмотр фактов, подтверждение изменений, удаление ошибок
- Observability: traces, журнал tool calls, журнал ошибок

### Не входит в MVP, но запланировано

- Интеграция с Level Kitchen: автоматический парсинг меню раз в неделю, точный подсчёт БЖУ по каталогу блюд (итерация 5)
- Несколько агентов (multi-agent)
- Двухуровневая модель (GPT-4o-mini + GPT-5.4)
- Голосовой интерфейс
- Веб-панель
- Распознавание еды по фото
- Другие wearable-интеграции
- Multi-user модель

## 5. Функциональные требования

### 5.1 Telegram

- Приём входящих сообщений (команды + свободный текст)
- Связь Telegram user id ↔ внутренний user id
- Текстовые ответы
- Proactive-сообщения (ежедневные, еженедельные)
- Идемпотентная обработка повторных доставок (без дублей)
- Webhook в production, polling в dev

### 5.2 Пользовательские сценарии

| Сценарий | Описание |
|----------|----------|
| Лог сна | Ручная запись + сопоставление с WHOOP |
| Лог питания | Время, продукты, калории, белок, комментарий. Несколько уровней точности |
| Лог тренировки | Тип, длительность, интенсивность, комментарий. Ручной + из WHOOP |
| Ежедневная рекомендация | На основе recovery, сна, тренировок, питания |
| Недельная сводка | Сон, recovery, тренировки, режим, паттерны за 7 дней |
| Корректировка профиля | Цель, ограничения, режим, предпочтения |
| Объяснение рекомендации | По запросу — на каких данных основан вывод |

### 5.3 WHOOP интеграция

- OAuth 2.0 с offline scope
- Данные: sleep, recovery, workout, cycles
- Синхронизация: periodic + webhooks
- Token refresh ~раз в час, защита от конкурентных refresh
- Нет доступа к continuous heart rate (ограничение WHOOP API)
- Ручная перепривязка и отзыв доступа

### 5.4 Память — 4 слоя

| Слой | Содержание |
|------|-----------|
| Профиль | Устойчивые факты: цель, ограничения, предпочтения, антропометрия |
| Текущее состояние | Агрегаты: вес, средний сон за неделю, последний recovery |
| Журнал событий | Сырые записи: сон, еда, тренировки, заметки |
| Выведенные правила | Гипотезы с confidence: влияние кофе на сон и т.п. |

Двухфазная модель: запись → consolidation в устойчивую память.

### 5.5 Логика обновления памяти

- Одноразовые события → журнал
- Устойчивые факты → профиль только после подтверждения пользователя или повторного подтверждения через несколько событий
- Выводы агента → гипотезы с confidence, временем, статусом
- Дедупликация, конфликт-резолвинг, забывание устаревших фактов

### 5.6 Tools агента (минимальный набор)

- `get_user_profile` — профиль пользователя
- `get_current_state` — текущие агрегаты
- `get_recent_logs` — последние записи по типу
- `save_sleep_log` — запись сна
- `save_meal_log` — запись питания
- `save_workout_log` — запись тренировки
- `save_note` — заметка / самочувствие
- `save_derived_rule` — гипотеза агента
- `list_memory_candidates` — кандидаты на устойчивую память
- `promote_memory_candidate` — перенести кандидата в профиль
- `delete_memory_item` — удалить факт
- `get_week_summary` — недельная сводка
- `sync_whoop_data` — ручная синхронизация WHOOP
- `get_latest_whoop_metrics` — последние метрики WHOOP

Memory distillation — отдельный tool, не прямая запись из каждого ответа.

## 6. Нефункциональные требования

### Надёжность
- Идемпотентность обработки Telegram updates и WHOOP webhooks
- Retry policy для внешних API
- Обработка WHOOP ошибок: 401, 429, 500
- Журнал ошибок и алерты

### Производительность
- Разумное время ответа на сообщение
- Разделение быстрых ответов и тяжёлых фоновых задач
- Selective memory injection (не весь контекст в prompt)
- Trimming и compression сессий

### Observability
- Traces по вызовам агента
- Лог tool calls с аргументами и результатами
- Анализ ошибок и деградации качества

### Безопасность
- API-ключи и токены в secret storage
- Токены не логируются в открытом виде
- Отзыв WHOOP-доступа пользователем
- Удаление пользовательских данных
- Доступ к данным ограничен по user id

### Расширяемость
- Модель OpenAI конфигурируемая
- Tools расширяемые
- Telegram — один из каналов, не единственный
- Memory layer независим от LLM

## 7. Архитектура

### Компоненты

- **Telegram Adapter** — приём/отправка сообщений
- **Application Backend** — маршрутизация, вызов агента, orchestration, retries, планировщик
- **OpenAI Agent Layer** — Agents SDK, system instructions, tools, traces
- **Persistence Layer** — PostgreSQL, таблицы для событий/профиля/памяти/sync
- **WHOOP Integration** — OAuth, token refresh, webhooks, periodic sync
- **Scheduler** — ежедневные сообщения, token refresh, nightly sync, weekly summaries

### Стек

- Python
- FastAPI
- PostgreSQL
- OpenAI Python SDK + Agents SDK
- APScheduler (фоновые задачи в MVP)
- Docker

### Сущности БД

`users`, `telegram_accounts`, `whoop_connections`, `user_profile`, `sleep_logs`, `recovery_logs`, `workout_logs`, `meal_logs`, `body_metrics`, `daily_notes`, `memory_notes`, `derived_rules`, `weekly_summaries`, `agent_runs`, `tool_calls`, `sync_events`

Каждая запись имеет `source`: user_manual | whoop_api | agent_inferred | system_aggregated.
Inferred-записи: confidence, created_at, updated_at, status, confirmation flag.
Внешние записи: external_id + last_synced_at.
Soft delete на логах и памяти.

## 8. Требования к агенту

- Опирается на structured context из tools, не на предположения
- Не делает долговременных выводов после одного события
- Различает: факт, наблюдение, гипотезу, рекомендацию
- Признаёт недостаточность данных
- Объясняет, почему дал совет
- Сжатый понятный стиль
- Не формулирует как медицинский диагноз
- Disclaimer при высокорисковых рекомендациях

## 9. Quality layer (детерминированные правила в коде)

- Нет recovery за сегодня → не делать уверенных выводов о нагрузке
- Сон < порога 3 дня подряд → повышенная осторожность
- Смена цели → пересчёт анализа с новой даты
- Валидация данных (калории, длительность)
- Приоритет источников (WHOOP > manual для sleep metrics)
- Fallback при ошибке модели или tool execution

## 10. Критерии готовности MVP

- [ ] Пользователь может подключить Telegram-бота
- [ ] Пользователь может подключить WHOOP
- [ ] Автоматический импорт sleep, recovery, workouts из WHOOP
- [ ] Ручное логирование питания и комментариев
- [ ] Ежедневный ответ на основе текущего состояния
- [ ] Weekly summary
- [ ] Память в БД, не в истории чата
- [ ] Изменение устойчивых фактов требует подтверждения
- [ ] Traces и технические логи
- [ ] Прозрачность: какие tools вызывались, какие данные использованы

## 11. Ключевые риски

| Риск | Митигация |
|------|----------|
| Контекстное загрязнение (вся память в prompt) | Selective injection, trimming, compression |
| Ложная консолидация памяти | Two-phase memory, подтверждение пользователя |
| Конкурентный refresh WHOOP токенов | Lock на refresh операцию |
| Логика только в LLM (не повторяемо) | Детерминированные правила в коде |
| Смешение чата и памяти | Жёсткое разделение: переписка ≠ журнал ≠ профиль ≠ rules |
